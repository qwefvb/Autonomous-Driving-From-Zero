# 5.1 大语言模型（LLM）基础

在进入 `5.3_VLM` 之前，先学这一节并不是为了把自动驾驶硬拗成一个“文本任务”，而是因为后续的视觉语言模型（VLM）本质上是把**视觉特征**和**语言建模能力**接到了一起。换句话说，VLM 之所以能“看图说话”、做场景解释、理解驾驶指令，背后依赖的正是 NLP 和 LLM 这条技术路线积累下来的表示学习、上下文建模和生成能力。

所以，这一节的主线很明确：

> **我们学习 LLM，不是为了单独做文本任务，而是为了理解后续 VLM 的语言侧基础。**

如果不先理解 `token`、`embedding`、`Transformer`、`pretraining`、`BERT / GPT` 这些概念，那么到了 VLM 章节时，很多结构都会变成“记名词”，而不是“真懂原理”。

> [!TIP]
> **把这一章看成 VLM 的语言预备课。**  
> 后面的 VLM 并不是凭空多出一个“会看图的 AI”，而通常是“视觉编码器 + 语言模型”拼起来的系统。你现在学的，就是其中“语言模型”这一半。

---

## 1. 为什么在 VLM 之前必须先学 LLM

传统自动驾驶系统的核心输入当然不是聊天文本，而是相机、激光雷达、地图、预测轨迹等结构化或半结构化信息。但在大模型路线里，研究者越来越希望模型具备下面这些能力：

- 不只是“看见目标”，还要**解释场景**。
- 不只是“输出决策”，还要**用语言说明理由**。
- 不只是“执行固定接口”，还要**理解人类指令或高层任务描述**。
- 不只是“提取视觉特征”，还要把视觉信息接入一个已经具备通用知识与推理能力的模型。

这正是 VLM 出现的重要背景。它通常不是重新从零发明一种完全不同的智能，而是沿着下面这条路线发展出来的：

`自然语言处理（NLP） -> 预训练语言模型（PLM） -> 大语言模型（LLM） -> 视觉语言模型（VLM）`

其中：

- `NLP` 解决的是“机器如何表示和处理语言”。
- `预训练语言模型` 解决的是“如何先从海量文本里学到通用语言知识”。
- `LLM` 解决的是“如何把这种语言能力做大、做通用、做成基础模型”。
- `VLM` 则是在这个基础上，把图像或视频特征接入语言模型。

因此，从学习顺序上看，**VLM 的前置知识并不是传统计算机视觉本身，而是“语言模型究竟是怎么工作的”**。

## 2. 必要的 NLP 前置概念

这一节不把 NLP 写成一门独立完整课程，而只保留理解 LLM 必须用到的最核心概念。

### 2.1 什么是 NLP

`NLP`（Natural Language Processing，自然语言处理）研究的是：如何让计算机表示、理解、生成人类语言。

人类眼里的语言是“词、句子、语义、上下文”，但计算机本身只能处理数字。因此，NLP 的第一个基本问题永远是：

> **如何把语言变成模型能计算的对象。**

这也就是后面 `token`、`vocabulary`、`embedding` 等概念存在的原因。

### 2.2 token、vocabulary、sequence

大语言模型不会直接“看懂一句话”，它首先会把文本切成更小的离散单元，这些单元叫做 `token`。

- 一个 `token` 可能是一个字、一个词、一个词片段，甚至一个符号。
- 所有可用 token 的集合，叫做 `vocabulary`（词表）。
- 一段文本在模型看来，本质上是一个 `token sequence`（token 序列）。

例如一句话：

`前方有一辆减速的车`

在人类看来是一个完整语义单元，但模型通常会先把它拆成若干 token，然后再处理这些 token 之间的关系。

这里最重要的理解是：

- **语言模型先处理的是“离散符号序列”，不是直接处理“语义本身”。**
- 模型所谓的“理解能力”，是建立在大量序列统计规律和表示学习之上的。

### 2.3 one-hot 与 embedding

如果只想让计算机区分不同 token，最朴素的方法是 `one-hot` 编码：词表里每个 token 对应一个维度，当前 token 的位置记为 1，其余位置全是 0。

这种表示的问题很明显：

- 维度极高；
- 稀疏；
- 无法表达词与词之间的相似性；
- “汽车”和“卡车”在 one-hot 空间里并不会比“汽车”和“星期三”更接近。

于是就有了 `embedding`（词嵌入/向量表示）：

- 每个 token 不再表示成一个巨大的稀疏向量；
- 而是表示成一个低维、稠密、可学习的向量；
- 相近语义的 token，往往会在向量空间里更接近。

`Word2Vec` 是这条路线上的经典工作之一，它推动了“用连续向量表示词语”的思路普及开来，可参考 Mikolov 等人的论文 [Efficient Estimation of Word Representations in Vector Space](https://arxiv.org/abs/1301.3781)。

可以把 `embedding` 粗略理解成：

> **token 的“机器语义坐标”。**

当然，这个说法并不严格，因为向量里并没有单独一个维度写着“语义 = 3.2”。但它足够帮助初学者建立直觉：模型不是在死记单词编号，而是在学习一种可计算的语义表示。

### 2.4 context、language model 与 next-token prediction

仅有 token 和 embedding 还不够，因为同一个词在不同上下文里可能含义不同。

例如：

- “模型很大”
- “道路模型”
- “世界模型”

这里“模型”的含义就依赖上下文。于是 NLP 的第二个核心问题出现了：

> **如何根据上下文理解当前 token。**

这就引出了 `context`（上下文）和 `language model`（语言模型）。

所谓语言模型，本质上是在建模：

- 一个 token 后面最可能出现什么；
- 或者说，给定前文，下一步最可能生成哪个 token。

这也是今天大多数生成式 LLM 的核心训练目标：`next-token prediction`（下一个 token 预测）。

初学者很容易误解为：

> “LLM 不就是在猜下一个词吗？”

这句话不完全错，但不够准确。更完整的理解是：

- 训练目标表面上是“预测下一个 token”；
- 但为了把这件事做对，模型被迫学习语法、事实、常识、上下文依赖、风格模式，甚至一定程度上的推理结构。

也就是说，**下一个 token 预测是训练入口，不是能力上限。**

## 3. 从词向量、RNN 到 Transformer 的演进

### 3.1 从“表示词”到“表示序列”

早期 NLP 很多时候先解决“如何表示词”，再解决“如何表示句子”。词向量让 token 有了更好的数值表示，但句子是一个**有顺序的序列**，所以模型还需要考虑：

- token 的先后顺序；
- 前后依赖关系；
- 长距离上下文；
- 句子整体含义。

这就把问题从“单个词表示”推进到了“序列建模”。

### 3.2 RNN：让模型按顺序读文本

`RNN`（Recurrent Neural Network，循环神经网络）是一类经典序列模型。它的直觉很简单：

- 依次读入一个个 token；
- 每读一个 token，就更新一次内部状态；
- 当前时刻的状态同时依赖“当前输入”和“之前记住的内容”。

所以，RNN 很像“边读边记”的模型。

它的重要意义在于：第一次比较自然地把“上下文”引入了神经网络语言处理中。模型不再只看当前 token，而是尝试把过去的信息压进一个隐藏状态里。

### 3.3 LSTM / GRU：缓解长程依赖问题

标准 RNN 很快遇到一个经典困难：**长程依赖**。

如果一句话很长，模型早期读到的信息在后面容易逐渐丢失，训练时也容易出现梯度消失或梯度爆炸。于是研究者提出了 `LSTM` 和 `GRU` 这类改进结构，其中 `LSTM` 是代表性工作，可参考 Hochreiter 与 Schmidhuber 的经典论文 [Long Short-Term Memory](https://www.bioinf.jku.at/publications/older/2604.pdf)。

`LSTM` 的核心直觉不是“更复杂”，而是“更会记和忘”：

- 有些信息要长期保留；
- 有些信息只需要短暂使用；
- 有些旧信息需要被遗忘，以免干扰当前理解。

从教学角度看，RNN/LSTM 带来的最大启发是：

> **语言理解不是只看当前 token，而是不断维护一个上下文状态。**

### 3.4 Seq2Seq：从“读一句”到“生成一句”

当研究目标从分类转向翻译、摘要、对话等任务时，只会“读一句”还不够，模型还要“生成另一句”。这催生了 `Encoder-Decoder` 范式，也就是常说的 `Seq2Seq`（sequence to sequence）。

典型思路是：

- `Encoder` 负责把输入序列编码成内部表示；
- `Decoder` 负责基于这个表示生成输出序列。

这条路线在机器翻译上非常重要，可参考 Sutskever 等人的 [Sequence to Sequence Learning with Neural Networks](https://arxiv.org/abs/1409.3215)。

这一步非常关键，因为后面很多大模型架构讨论里都会反复出现：

- `Encoder-only`
- `Decoder-only`
- `Encoder-Decoder`

它们都可以追溯到这类问题设定。

### 3.5 为什么 RNN 路线最终不够

虽然 RNN/LSTM 在历史上非常重要，但它们逐渐暴露出几个限制：

- 必须按顺序处理，难以高效并行；
- 长文本上下文仍然难处理；
- 即使有注意力机制辅助，信息路径仍然相对较长；
- 当模型和数据规模继续扩大时，训练效率和表达能力都越来越受限。

于是，NLP 进入了下一个决定性阶段：`Transformer`。

## 4. Transformer 为什么改变了 NLP

### 4.1 Self-Attention 的核心思想

`Transformer` 的代表性论文是 Vaswani 等人的 [Attention Is All You Need](https://arxiv.org/abs/1706.03762)。这篇工作最重要的贡献，不只是提出一个新模块，而是把序列建模的主角从“循环”换成了“注意力”。

所谓 `self-attention`，可以先直观理解成：

> **处理某个 token 时，不再只依赖一个按时间传递的隐藏状态，而是直接去看序列里哪些位置和自己最相关。**

例如在一句长句里，模型处理某个词时，可能需要重点关注：

- 主语是谁；
- 修饰词落在哪个对象上；
- 前面哪一段决定了当前词的含义；
- 远处哪个 token 和当前 token 在语义上强相关。

`self-attention` 的价值就在于，它允许模型**动态选择关注对象**，而不是被迫把所有信息都压进一个单一的循环状态里。

如果你想要一个非常直观的可视化讲解，Jay Alammar 的博客 [The Illustrated Transformer](https://jalammar.github.io/illustrated-transformer/) 仍然是非常好的入门材料。

### 4.2 Query、Key、Value 不必神秘化

很多初学者第一次看到 `Query / Key / Value` 会立刻觉得抽象。其实可以先用一个不那么数学、但足够有用的理解：

- `Query`：当前 token 想“找什么信息”
- `Key`：其他 token 各自“能提供什么索引”
- `Value`：真正要被取回的内容

当一个 token 发出自己的 `Query` 后，它会和所有 token 的 `Key` 计算相关性，决定注意力权重，再对这些 token 的 `Value` 做加权汇总。

所以，`self-attention` 的本质不是“神秘公式”，而是：

> **让每个 token 学会去全局检索和聚合和自己最相关的信息。**

### 4.3 Transformer 相对 RNN 的优势

Transformer 之所以迅速统治 NLP，一个重要原因是它同时解决了“效果”和“规模”两个问题。

它的主要优势包括：

- 可以并行处理整段序列，而不是一步一步递推；
- 更容易捕捉长距离依赖；
- 在大规模数据和大参数规模下更容易训练；
- 结构统一，便于扩展到更大模型、更长上下文和更多模态。

今天的 LLM，本质上几乎都建立在 Transformer 路线之上。换句话说：

> **没有 Transformer，就很难有今天意义上的 LLM。**

### 4.4 Encoder-only、Decoder-only、Encoder-Decoder

理解后续 `BERT`、`GPT`、`T5` 这类模型时，最先要搞清楚的就是架构范式。

#### `Encoder-only`

特点是：

- 输入整段文本；
- 双向建模上下文；
- 更擅长理解类任务。

典型代表是 `BERT`。它更适合：

- 文本分类；
- 序列标注；
- 阅读理解；
- 检索表示学习。

#### `Decoder-only`

特点是：

- 按生成方向逐步预测下一个 token；
- 天然适合自回归生成；
- 是今天主流 LLM 最常见的骨架。

典型代表是 `GPT` 系列。它更适合：

- 长文本生成；
- 对话；
- 代码生成；
- 通用指令跟随。

#### `Encoder-Decoder`

特点是：

- `Encoder` 读输入；
- `Decoder` 生成输出；
- 特别适合输入输出都很明确的转换任务。

典型场景包括：

- 机器翻译；
- 摘要；
- 文本改写。

把三者放在一起看，最重要的不是死记名字，而是理解：

- **BERT 更偏“理解器”**
- **GPT 更偏“生成器”**
- **Encoder-Decoder 更偏“转换器”**

## 5. 预训练语言模型与 LLM 的形成

### 5.1 什么是预训练

现代语言模型的一次根本转变，是从“只为单个任务训练一个模型”，转向“先在海量通用文本上预训练，再迁移到具体任务”。

这就是 `pretraining`（预训练）。

它的核心逻辑是：

- 大量无标注文本远比高质量人工标注便宜；
- 语言本身包含巨量统计规律、知识关联和表达模式；
- 如果先让模型在海量文本上学习通用语言结构，那么它在下游任务上就能更快适应。

这类模型常被称为 `PLM`（Pretrained Language Model，预训练语言模型）。

### 5.2 BERT：双向理解模型的代表

`BERT` 是预训练语言模型时代的代表性里程碑之一，可参考 Devlin 等人的 [BERT: Pre-training of Deep Bidirectional Transformers for Language Understanding](https://arxiv.org/abs/1810.04805)。

它的重要性在于：

- 它把 `Transformer encoder` 做成了通用语言理解骨架；
- 它强调双向上下文建模；
- 它证明了“预训练 + 微调”在大量 NLP 任务上非常有效。

如果说 Word2Vec 解决的是“词如何表示”，RNN/Seq2Seq 解决的是“序列如何建模”，那么 BERT 更像是在说：

> **我们可以先训练一个通用语言表示器，再把它迁移到很多任务上。**

### 5.3 GPT：从语言模型到生成式基础模型

与 BERT 不同，`GPT` 路线更强调 `decoder-only` 的自回归生成能力，也就是根据前文持续预测后续 token。

这条路线后来之所以发展成 LLM 主流，是因为它非常契合几个趋势：

- 训练目标统一；
- 生成能力自然；
- 扩展到超大数据和超大参数时效果持续提升；
- 更适合做通用交互接口。

所以今天人们提到 LLM，很多时候默认想到的就是以 `GPT / LLaMA / Qwen` 为代表的 `decoder-only Transformer` 大模型。

### 5.4 什么是“大语言模型”

“大语言模型”不只是“参数很多的语言模型”。

更准确地说，`LLM` 通常同时具备以下特征：

- 基于大规模语料预训练；
- 采用 Transformer 架构；
- 参数规模显著扩大；
- 上下文建模能力增强；
- 具备较强的迁移能力和通用任务适应能力；
- 可以通过提示（prompt）直接完成多种任务。

所以，`LLM` 是多个因素共同放大的结果：

- 更大的模型
- 更多的数据
- 更长的上下文
- 更强的算力
- 更成熟的训练与对齐方法

这也解释了为什么“参数变大”虽然重要，但不是唯一答案。

## 6. LLM 的核心能力、局限与工程要点

### 6.1 LLM 到底强在哪里

当模型规模足够大、训练数据足够多时，LLM 往往会表现出非常强的通用能力，包括：

- 文本生成；
- 摘要与改写；
- 问答；
- 信息抽取；
- 多轮对话；
- 一定程度的推理与规划；
- 代码生成；
- 把自然语言当作统一接口来驱动任务。

这也是它对后续 VLM 特别关键的原因：  
一旦图像特征能被接入这个系统，模型就不再只是“识别图里有什么”，而是有机会把视觉内容纳入一个更通用的理解与生成框架。

### 6.2 指令微调与对齐

光有预训练还不够。预训练模型擅长续写文本，但不一定擅长“听懂人话并按要求完成任务”。于是就有了：

- `instruction tuning`（指令微调）
- `alignment`（对齐）

它们的目标是让模型从“会续写”变成“会配合使用者完成任务”。

一个常见理解框架是：

1. **预训练**：让模型学会语言与知识分布。
2. **指令微调**：让模型学会按任务描述来回答。
3. **对齐**：让模型输出更符合人类偏好、更安全、更稳定。

这里常见的方法包括监督微调，以及基于偏好数据的对齐方法。GPT-4 技术报告 [GPT-4 Technical Report](https://arxiv.org/abs/2303.08774) 可以作为理解现代大模型能力边界的一个代表性参考。

### 6.3 LLM 的典型局限

虽然 LLM 很强，但它绝不是“全知全能的智能体”。最常见的局限包括：

- **幻觉**：模型会生成看似合理但实际错误的内容。
- **知识时效性有限**：训练后不会自动持续更新世界知识。
- **上下文长度有限**：即使上下文窗口越来越长，也不是无限的。
- **可靠性不稳定**：同一问题换个问法，结果可能变化明显。
- **缺乏真实世界 grounding**：如果没有外部感知、检索或工具，它对现实世界的理解仍然间接。

初学者还常有一个误解：

> **“模型说得很像真话，所以它就真的理解了。”**

这是一种非常危险的错觉。LLM 的强项是概率建模与模式生成，不等于它天然具备严格、可验证、可追责的现实理解能力。

### 6.4 为什么这些局限在自动驾驶里尤其敏感

在自动驾驶或车端系统里，上述局限会被放大：

- 幻觉不能只是“答错一道题”，它可能误导用户理解系统状态。
- 不稳定输出会破坏系统可预测性。
- 语言模型擅长“说”，但安全系统要求的是“可验证地做对”。
- 纯语言推断无法替代实时感知、几何约束和安全规则。

因此，在自动驾驶语境里，LLM 更适合承担：

- 场景解释；
- 驾驶日志总结；
- 人机交互；
- 规划辅助分析；
- 多模态理解接口。

而不应被简单理解为“直接替代整套安全关键驾驶主链路”。

> [!TIP]
> **这也是为什么后续 VLM 很有价值，但又不能被神化。**  
> 它能让系统“看得懂并说得出来”，但不意味着它天然满足自动驾驶对实时性、鲁棒性、可验证性和安全性的全部要求。

---

## 7. 从 LLM 走向 VLM：视觉信息是如何接进来的

现在可以回答一个非常关键的问题：

> **VLM 为什么不是凭空出现的？**

因为它通常就是在 LLM 基础上，把视觉特征作为额外输入接进来。

一个非常实用的理解框架是：

1. **视觉编码器**负责把图像或视频变成特征。
2. **投影层 / 桥接模块**负责把视觉特征映射到语言模型可处理的表示空间。
3. **LLM** 负责基于这些表示做语言理解、问答、解释和生成。

也就是说，VLM 的关键并不是“让 LLM 直接看像素”，而是：

- 先由视觉模块提取视觉表征；
- 再把这些表征对齐到语言模型的输入空间；
- 然后利用 LLM 强大的上下文建模和生成能力来处理多模态任务。

这也解释了后面一些典型路线为什么成立：

- `CLIP` 更强调图文对齐；
- `BLIP / BLIP-2` 更强调视觉与语言之间的高效交互；
- `LLaVA` 一类方法更像是“给 LLM 接上一双眼睛”。

从这个角度再看，`5.3_VLM` 其实是在回答：

- 视觉特征怎么提；
- 图像和语言怎么对齐；
- 多模态输入如何接入 LLM；
- 为什么模型能基于图像回答问题、解释场景、执行视觉指令。

所以，本节和下一节的关系非常自然：

- **这一节解决“语言模型这一半是怎么来的”。**
- **下一节解决“视觉这一半是怎么接上去的”。**

## 8. 学习路径与延伸阅读

### 8.1 推荐学习顺序

如果你的目标是理解自动驾驶里的 VLM，而不是单独深挖传统 NLP，全章最推荐的学习顺序是：

1. 先理解 `token -> embedding -> context -> language model` 这条最小链路。
2. 再理解为什么 `RNN/LSTM` 不足以支撑今天的大模型规模。
3. 再把 `Transformer` 看懂，尤其是 `self-attention`、`encoder`、`decoder`。
4. 然后理解 `BERT` 与 `GPT` 的分野。
5. 最后带着“视觉特征如何接到语言模型里”的问题进入 `5.3_VLM`。

### 8.2 文内提到的核心参考

- [Efficient Estimation of Word Representations in Vector Space](https://arxiv.org/abs/1301.3781)

### 8.3 小结

读完这一节后，最重要的不是背下多少模型名，而是把下面这条链路真正串起来：

1. 文本先被切成 `token`。
2. token 被映射成 `embedding`。
3. 模型需要根据 `context` 建模序列关系。
4. `Transformer` 用注意力机制高效处理长上下文。
5. 大规模 `pretraining` 让语言模型获得通用能力。
6. `LLM` 在此基础上成为通用生成与理解骨架。
7. `VLM` 再把视觉特征接入这个骨架。

当你把这条链路想清楚后，再去看后面的 `CLIP`、`BLIP`、`LLaVA`，就不会只是在记结构图，而是在理解：

> **为什么“语言模型”会成为多模态系统里的核心大脑之一。**
