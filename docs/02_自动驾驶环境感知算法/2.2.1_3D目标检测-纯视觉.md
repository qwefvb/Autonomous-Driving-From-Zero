# 2.2.1 3D目标检测-纯视觉

纯视觉 3D 目标检测（Vision-only 3D Object Detection）是自动驾驶感知中最具挑战、也最具代表性的任务之一。它要求系统**仅依赖相机图像**，就恢复出周围目标在真实三维世界中的位置、尺寸、朝向，必要时还包括速度等运动属性。

如果说 2D 检测回答的是：

- 图中出现了什么目标？
- 这些目标在图像上大致位于哪里？

那么纯视觉 3D 检测要进一步回答：

- 这个目标在真实世界的什么位置？
- 它离车多远？
- 它朝向哪里？
- 它的长宽高大概是多少？
- 它是静止的，还是正在运动？

**因此，纯视觉 3D 检测本质上是在“只有投影结果、没有直接深度”的前提下，从 2D 图像反推出 3D 结构。**

它的标准输出通常是一组 3D 框参数：

- 目标中心位置 $\left(x, y, z\right)$
- 目标尺寸 $\left(w, l, h\right)$
- 目标朝向 `yaw`
- 目标类别 `class`
- 置信度 `score`
- 可选速度 $\left(v_x, v_y\right)$

这些结果会继续送往多目标跟踪、行为预测、风险评估、规划控制等下游模块。也正因为如此，纯视觉 3D 检测不是“图像理解的小升级”，而是自动驾驶从“看见画面”走向“理解空间”的关键一步。

> [!TIP]
> **可以把纯视觉 3D 检测理解为：**
> 2D 检测是在图上“圈出目标”，而 3D 检测是在真实道路坐标系里“摆出目标”。

---

## 1. 纯视觉 3D 检测在自动驾驶中的职责

自动驾驶中的 3D 检测最常面向以下几类交通参与者与障碍物：

- 动态目标：车辆、行人、骑行者、摩托车、公交车、卡车。
- 道路障碍物：锥桶、路障、施工设备、异常停靠车辆。
- 可运动目标：推车、婴儿车、低速工程车、特殊异形车。

相比 2D 检测，3D 检测在自动驾驶里的价值更加直接：

- 为跟踪提供物理空间中的初始状态。
- 为预测模块提供位置、朝向、速度等运动先验。
- 为规划控制提供可直接用于避障和路径决策的几何对象。
- 为风险评估提供“目标离我多远、是否正在靠近”的关键信息。

它和 2D 检测最本质的区别在于：**2D 检测仍停留在图像坐标系，3D 检测则必须进入车辆所在的物理世界坐标系。**

对于自动驾驶而言，这种区别并不是“精度更高一点”那么简单，而是能力边界的变化：

- 2D 框无法直接表示真实距离。
- 2D 框无法稳定支持速度估计。
- 2D 框很难直接用于轨迹规划。
- 3D 框则天然与车辆坐标系、BEV（Bird's Eye View，鸟瞰图）空间和运动模型兼容。

但纯视觉路线也有天然困难：

- 相机没有 LiDAR 那样的直接深度测量。
- 图像中的尺度变化和真实距离高度耦合。
- 远距离小目标、遮挡目标和夜间目标尤其难。
- 多相机系统还要处理跨视角对齐和外参误差问题。

因此，纯视觉 3D 检测的发展史，本质上就是在回答一个问题：

> 在缺少直接深度观测的前提下，如何尽量稳定、尽量高效地恢复真实三维世界？

## 2. 基础概念与任务工作流

### 2.1 一个纯视觉 3D 检测器到底在做什么

一个完整的纯视觉 3D 检测器，通常同时在做 5 件事：

1. **图像特征提取**：从单目图像或多相机环视图像中提取语义特征。
2. **空间结构恢复**：显式或隐式地推断深度、尺度和空间布局。
3. **3D 表示构建**：把图像特征转换成适合检测的 3D / BEV / query 表示。
4. **目标级推理**：预测类别、3D 框、朝向和可选速度。
5. **后处理与坐标变换**：把结果统一到车辆坐标系或 BEV 坐标系。

从宏观角度看，纯视觉 3D 检测大致可以抽象成下面这条链路：

```mermaid
graph LR
    A[输入图像: 单目或多相机] --> B[Backbone 提取图像特征]
    B --> C[空间恢复: 深度/几何/Query/BEV]
    C --> D[3D表示: Camera View / BEV / Sparse Queries]
    D --> E[Detection Head 预测3D框]
    E --> F[坐标统一与后处理]
    F --> G[最终3D目标]
```

和 2D 检测相比，纯视觉 3D 检测多出的关键难点是：

- 目标分类只是第一步。
- 目标定位不再是图像平面定位，而是三维空间定位。
- 框回归不再是 $(x, y, w, h)$，而是 $\left(x, y, z, w, l, h, yaw\right)$。

这意味着它必须同时处理：

- **语义识别（Classification）**
- **三维定位（3D Localization）**
- **姿态估计（Orientation Estimation）**
- **尺度估计（Size Estimation）**
- **可选运动估计（Velocity Estimation）**

### 2.2 3D 框、坐标系与投影关系

纯视觉 3D 检测里，最先要搞清楚的是：**同一个目标，可能同时存在于多个坐标系里。**

常见坐标系包括：

- **图像坐标系（Image Coordinate）**：目标表现为 2D 像素位置。
- **相机坐标系（Camera Coordinate）**：目标相对某个相机的位置。
- **车辆坐标系（Ego Coordinate）**：目标相对自车中心的位置。
- **BEV 坐标系（Bird's Eye View Coordinate）**：从上往下看的地面平面坐标。

在针孔成像模型下，三维点与图像点的投影关系可写成：

```math
s \begin{bmatrix}
u \\
v \\
1
\end{bmatrix}
=
K \begin{bmatrix}
X_c \\
Y_c \\
Z_c
\end{bmatrix}
```

其中：

- $\left(u, v\right)$ 是像素坐标。
- $\left(X_c, Y_c, Z_c\right)$ 是相机坐标系下的三维点。
- $K$ 是相机内参矩阵。
- $s$ 与深度相关。

这条公式非常重要，因为它揭示了纯视觉 3D 检测的根本难点：

> 图像只给出了投影结果，但没有直接给出深度 $Z_c$。

同一个图像框可能对应：

- 一个近处的小车
- 一个远处的大车
- 一个被部分遮挡、形状不完整的目标

因此，3D 检测里经常会出现下面这些参数化方式：

- **中心点参数化**：预测 3D 中心点 + 尺寸 + 朝向。
- **角点参数化**：预测 8 个 3D 角点，再恢复框体。
- **BEV 参数化**：在地平面上预测中心、长宽和朝向，再回归高度。
- **query 参数化**：用少量候选实例表示目标，在迭代推理中不断精修。

### 2.3 单目、环视图与四种核心表示范式

从输入角度看，纯视觉 3D 检测大致有两类主设定：

- **单目 3D 检测（Monocular 3D Detection）**：使用单张前视图像。
- **环视图 3D 检测（Surround-view 3D Detection）**：使用多相机环视图像，常见是 6 路或 8 路相机。

单目方法的优点是：

- 结构简单
- 成本低
- 更适合作为入门和前视感知基线

但它受限也很明显：

- 深度高度不确定
- 视野不完整
- 难以在车辆周围构建一致空间表示

因此，自动驾驶的主流纯视觉 3D 检测研究逐渐从单目前视走向**环视图 + BEV / Query 空间建模**。

从“空间表示如何构造”这个角度，可以把当前主流方法分成 4 种核心范式：

| 范式 | 核心思想 | 直觉理解 | 代表问题 |
| :--- | :--- | :--- | :--- |
| **Dense** | 尽量把空间铺开，构造稠密 3D / BEV 特征 | 先建空间地图，再检测目标 | 几何表达强，但成本高 |
| **Sparse** | 只保留少量候选点、query 或实例槽位 | 直接找目标，不先把空间铺满 | 高效，但先验依赖强 |
| **Hybrid** | 稠密空间提供底座，稀疏实例做聚焦推理 | 先粗建空间，再细找目标 | 是现实折中路线 |
| **Temporal** | 用多帧补足单帧信息不足 | 用时间换几何与稳定性 | 对对齐和记忆设计要求高 |

> [!TIP]
> **这里的 Temporal 更像“能力层”，而不是完全独立于其他三类的方法。**
> 实际上，时序机制经常叠加在 Dense、Sparse 或 Hybrid 路线上，用来提升深度稳定性和跨帧连续性。

### 2.4 一个现代纯视觉 3D 检测器的工作流

从工程实现上看，一个现代环视纯视觉 3D 检测器的常见工作流可以写成：

1. 输入多相机图像。
2. 提取每个视角的图像特征。
3. 通过显式深度、隐式投影或 query 采样，把多视角特征变到统一空间。
4. 在统一空间中做目标级推理。
5. 输出 3D 框、朝向、速度等结果。

如果是 BEV 路线，通常流程更接近：

```mermaid
graph LR
    A[多相机图像] --> B[共享Backbone]
    B --> C[View Transform / Lift-Splat]
    C --> D[BEV Feature]
    D --> E[BEV Detection Head]
    E --> F[3D Boxes]
```

如果是 Sparse 路线，通常流程更接近：

```mermaid
graph LR
    A[多相机图像] --> B[图像特征]
    B --> C[Object Queries / Reference Points]
    C --> D[跨视角特征采样]
    D --> E[迭代更新实例状态]
    E --> F[3D Boxes]
```

这也为后面的方法分类埋下了伏笔：

- Dense 更像“**先有空间，再找目标**”
- Sparse 更像“**先有目标假设，再向图像取证**”
- Hybrid 更像“**先用空间托底，再用目标实例精修**”

## 3. 方法演进：为什么 3D 检测会这样发展

纯视觉 3D 检测的发展，不只是模型越来越大，而是围绕几个核心矛盾不断演化：

- 图像没有直接深度，怎么恢复空间？
- 单张图证据不足，怎么提高稳定性？
- 多相机视角很多，怎么做统一表示？
- 稠密建模更完整，但太重；稀疏建模更轻，但是否会丢掉关键信息？
- 目标检测是目标级任务，是否真的有必要对整个空间做全量建模？

从历史脉络看，大致经历了下面几步：

### 3.1 从单目前视 3D 估计出发

早期纯视觉 3D 检测大多从单目问题入手：

- 先做 2D 检测
- 再利用透视几何、尺寸先验或关键点约束恢复 3D 位置

这类方法的特点是：

- 容易理解
- 容易从 2D 检测迁移
- 但深度非常难稳

也就是说，单目 3D 检测更像是在回答：

> 能不能从一张图中“猜出”一个合理的 3D 解释？

### 3.2 从前视图走向统一空间表示

当系统进入自动驾驶场景后，问题变得更复杂：

- 车前、车侧、车后都要看
- 目标要统一到自车坐标系
- 目标运动和规划控制更关心俯视平面的几何关系

于是研究者开始意识到：

> 纯视觉 3D 检测不能一直停留在“每个视角单独做检测”，而是需要一个统一的空间表示。

这推动了 **BEV（Bird's Eye View）** 路线的兴起。

### 3.3 为什么会分化成 Dense、Sparse、Hybrid

BEV 路线虽然自然，但很快又遇到了新的问题：

- 如果把整个空间都建成稠密特征图，几何表达很强，但算力和显存开销很大。
- 但如果完全不建空间、只靠少量 query 推理，又可能缺少完整的全局上下文。

于是，环视图纯视觉 3D 检测逐渐分化成三条长期共存的路线：

1. **Dense 路线**  
   尽量恢复完整的 BEV / 3D 空间，再在空间上做密集检测。

2. **Sparse 路线**  
   不先铺满空间，而是直接维护少量稀疏实例或 query，让模型围绕这些候选目标采样图像证据。

3. **Hybrid 路线**  
   既承认空间底座很重要，又承认目标级推理更高效，因此把两者结合起来。

### 3.4 为什么 Temporal 会成为重要补强

即便有了 Dense 或 Sparse 表示，单帧纯视觉仍然有很多固有局限：

- 单帧深度估计容易抖动。
- 被遮挡目标可能这一帧看不清，但前一帧看得更清楚。
- 远距离目标纹理极弱，单帧证据往往不足。
- 运动目标的速度和朝向变化本来就天然跨帧。

因此，多帧时序融合逐渐从“锦上添花”变成“纯视觉 3D 检测的重要补强方向”。

可以用一句话概括这一演进逻辑：

> 单目解决“能不能做”，环视解决“能不能统一空间”，Dense / Sparse / Hybrid 解决“怎么建模更合适”，Temporal 则解决“如何让结果更稳、更连续”。

## 4. 单目 3D 检测：从几何约束到深度建模

单目 3D 检测通常是学习纯视觉 3D 感知的起点，因为它最直接暴露了这个任务的本质难点：**只有投影，没有直接深度。**

### 4.1 为什么单目 3D 检测这么难

从单张图像恢复 3D 信息，本身就是病态问题（ill-posed problem）：

- 同样大小的 2D 框，可能来自不同距离和不同真实尺寸的物体。
- 遮挡会破坏目标轮廓和几何边界。
- 相机俯仰、道路坡度和截断都会影响深度判断。

因此，单目方法往往会引入：

- 物体尺寸先验
- 地平面约束
- 关键点约束
- 深度分布建模
- 伪点云或 lifting 表示

### 4.2 基于 2D 检测与几何约束的方法

这一类方法的思路最直观：

- 先做 2D 检测
- 再利用几何关系恢复 3D 属性

典型做法包括：

- 利用 2D 框高度估计距离
- 结合类别尺寸先验回推深度
- 利用投影一致性约束优化 3D 框

这类方法的优点是：

- 结构清晰
- 易于从成熟的 2D 检测器迁移
- 几何意义比较直观

缺点也很明显：

- 过度依赖先验尺寸
- 对遮挡和截断敏感
- 深度误差容易被放大

> [!TIP]
> 早期很多单目 3D 检测工作，本质上还是“2D 检测 + 几何修正”。它们帮助研究者建立了任务框架，但也很快暴露出：**仅靠后验几何修补，不足以支撑高质量 3D 感知。**

### 4.3 基于关键点、中心点与姿态回归的方法

另一类重要路线是把 3D 检测转化为更结构化的目标表示问题。

典型思路包括：

- 预测 3D 包围盒的投影角点
- 预测 3D 中心点的图像投影
- 预测目标朝向、局部偏移和尺寸

例如 `CenterNet` 风格的方法会把目标表示为中心点，再回归额外属性；`FCOS3D` 一类方法则把 2D anchor-free 检测扩展到 3D 设定中。

这类方法的优点是：

- 表示形式更紧凑
- 不依赖复杂的 anchor 设计
- 更容易和现代检测框架对齐

缺点是：

- 关键点预测本身会受遮挡影响
- 中心点与深度误差耦合强
- 朝向和尺寸回归不稳定时会造成整体框畸变

> [!TIP]
> **论文链接**：
> - [FCOS3D: Fully Convolutional One-Stage Monocular 3D Object Detection](https://arxiv.org/abs/2104.10956)
> - [SMOKE: Single-Stage Monocular 3D Object Detection via Keypoint Estimation](https://arxiv.org/abs/2002.10111)

### 4.4 基于深度分布、伪点云与 lifting 的方法

研究者很快意识到：单目 3D 检测的核心困难在于深度，因此很多方法开始显式建模深度。

典型思路包括：

- 直接预测像素级或 RoI 级深度
- 预测离散深度分布而不是单一深度值
- 先估计深度，再把图像提升到伪点云或伪体素空间

这类方法的重要意义在于：

- 它把“猜 3D 框”转向“先恢复部分几何，再做检测”
- 为后续环视图中的 Lift-Splat / BEV 变换奠定了思想基础

其优点是：

- 深度建模更显式
- 更容易把图像与空间联系起来
- 对 BEV 路线形成了自然过渡

但它仍然有局限：

- 单目深度本身仍不稳定
- 伪点云质量强依赖深度预测
- 一旦前端深度偏差很大，后续 3D 检测也会跟着失真

> [!TIP]
> **论文链接**：
> - [Pseudo-LiDAR from Visual Depth Estimation: Bridging the Gap in 3D Object Detection for Autonomous Driving](https://arxiv.org/abs/1812.07179)
> - [PGD: Probabilistic and Geometric Depth: Detecting Objects in Perspective](https://arxiv.org/abs/2107.14160)

### 4.5 基于 Query / Transformer 的单目方法

随着 DETR 范式兴起，单目 3D 检测也开始引入 query-based 建模。

这类方法通常会：

- 使用一组 object queries
- 在图像特征上做跨位置的信息聚合
- 直接输出 3D 框参数

它们的价值不在于“Transformer 更时髦”，而在于：

- 目标级表示更统一
- 更方便后续扩展到多视角和时序设定
- 更容易与稀疏实例建模接轨

但原始问题并没有消失：

- 单目缺深度这一根本矛盾依旧存在
- query 设计、先验注入和收敛稳定性仍很关键

> [!TIP]
> **论文链接**：
> - [MonoDETR: Depth-guided Transformer for Monocular 3D Object Detection](https://arxiv.org/abs/2203.13310)

### 4.6 单目路线的意义与边界

单目 3D 检测的真正价值，不只是它本身能做出多强的结果，而是它把纯视觉 3D 问题的核心矛盾暴露得非常彻底：

- 你必须面对深度的不确定性
- 你必须决定是否显式建模几何
- 你必须决定是先建空间，还是直接做目标级推理

也正因为如此，单目路线最终把研究推向了更适合自动驾驶的方向：

- 从单相机走向环视多相机
- 从前视 3D 估计走向统一 BEV 空间
- 从单帧推理走向时序融合

## 5. 环视图 3D 检测：Dense、Sparse 与 Hybrid 三条主线

环视图 3D 检测是当前自动驾驶纯视觉感知研究的主战场。它的关键变化在于：**不再围绕单张前视图做 3D 估计，而是围绕整车周围的统一空间表示做检测。**

### 5.1 Dense BEV：先构造稠密空间，再在空间中检测

Dense 路线的核心思想是：

> 先把多相机图像提升到统一的空间表示，尤其是 BEV 表示，再在这个空间上做检测。

之所以 BEV 特别适合自动驾驶，是因为：

- 它天然统一了多相机视角。
- 它更符合车辆规划和控制所使用的地面平面几何。
- 它能让目标位置和朝向在物理世界中更稳定地表达。

#### 5.1.1 Dense BEV 的基本工作流

Dense BEV 方法通常包含以下步骤：

1. 用共享 backbone 提取多相机图像特征。
2. 为每个像素或每个 feature cell 估计深度分布，或通过隐式方式进行视图变换。
3. 把图像特征 lift 到三维空间，再 splat / pool 到 BEV 平面。
4. 得到统一的 BEV feature map。
5. 在 BEV 上运行 dense detection head。

如果采用显式深度分布，一个常见的 lifting 形式可以写成：

$$
F_{bev}(x, y) = \sum_{u, v, d} p(d \mid u, v)\, F_{img}(u, v)\, \mathbf{1}\left(\Pi(u, v, d) \rightarrow (x, y)\right)
$$

其中：

- $F_{img}(u, v)$ 是图像特征。
- $p(d \mid u, v)$ 是像素处的深度分布。
- $\Pi$ 表示从图像坐标和深度映射到 BEV 位置。

这条式子表达的直觉是：

> 每个图像特征不再只属于二维图像，而是按照一定深度概率，被“抬升”到物理空间中的不同位置。

#### 5.1.2 显式深度分布与 Lift-Splat 主线

`Lift-Splat-Shoot (LSS)` 是这条路线的代表性工作之一。

它的贡献在于把纯视觉环视 BEV 检测的基本范式讲得非常清楚：

- **Lift**：把图像特征沿深度维展开
- **Splat**：投影并聚合到 BEV 网格
- **Shoot**：在 BEV 上执行下游任务

`LSS` 的意义不仅在于性能，更在于它建立了一个非常直观的中间表示：

- 图像和 BEV 不再是完全断开的两种空间
- 深度分布成为连接图像与 BEV 的桥梁

> [!TIP]
> **论文链接**：[Lift, Splat, Shoot: Encoding Images from Arbitrary Camera Rigs by Implicitly Unprojecting to 3D](https://arxiv.org/abs/2008.05711)

#### 5.1.3 从 BEVDet 到 BEVDepth：Dense 路线的工程强化

后续很多工作围绕 LSS 路线持续强化。

`BEVDet` 的代表性在于：

- 它把纯视觉环视 BEV 检测做成了更实用、更工程化的框架
- 强调高效的 BEV 特征生成与检测头设计

`BEVDepth` 的关键推进在于：

- 更认真地对待深度质量
- 通过更强的深度监督或更合理的深度建模，提升 BEV 变换的几何可靠性

这类工作的共同点是：**承认深度质量对 Dense BEV 至关重要。**

因为只要图像特征被投错位置，后续 BEV 检测头再强，也是在错误空间上工作。

> [!TIP]
> **论文链接**：
> - [BEVDet: High-Performance Multi-Camera 3D Object Detection in Bird-Eye-View](https://arxiv.org/abs/2112.11790)
> - [BEVDepth: Acquisition of Reliable Depth for Multi-view 3D Object Detection](https://arxiv.org/abs/2206.10092)

#### 5.1.4 Dense BEV 的优势与局限

Dense 路线的优势很明确：

- 全局空间表示完整
- 多相机融合自然
- 对遮挡、远距离目标和局部缺失更稳
- 非常适合和 BEV 分割、在线建图、Occupancy 等任务共享中间表示

但它的代价也同样明确：

- 显存消耗大
- 计算量高
- 时延压力大
- 深度估计误差会直接污染 BEV 表示

可以把 Dense 路线理解为：

> 先花较大代价建立“场景底座”，再在这个底座上做目标检测。

| Dense BEV 的优点 | Dense BEV 的缺点 |
| :--- | :--- |
| 空间表示完整 | 计算和显存开销大 |
| 多视角融合自然 | 依赖高质量深度或视图变换 |
| 对 BEV 下游任务友好 | 端侧部署压力大 |
| 对复杂场景上下文建模强 | 稀疏目标任务上可能有冗余计算 |

### 5.2 Sparse：不先铺满空间，直接进行目标级建模

Sparse 路线的核心思想是：

> 既然检测本身是目标级任务，而大部分空间本来就是空的，那么是否可以不先构造完整稠密空间，而是直接维护少量候选实例，在这些实例上做推理？

这类方法的“稀疏”主要体现在：

- 候选对象数量少
- 只在少量 reference points 或 object queries 周围取证
- 不强求先生成一张完整 BEV 特征图

#### 5.2.1 Sparse 路线为什么会兴起

Dense BEV 虽然强，但很重；而检测本身只关心少量目标。因此研究者开始提出：

- 能不能直接用一组 queries 代表潜在目标？
- 能不能让每个 query 主动去多视角图像中采样证据？
- 能不能跳过对整张 BEV 的全量建模？

于是，环视图 3D 检测中的 Sparse 路线逐渐成形。

#### 5.2.2 常见的 Sparse 设计方式

常见的 Sparse 设计包括：

- **Object Queries**：用少量可学习 query 表示潜在目标。
- **Reference Points / 3D Anchors**：给 query 一个显式 3D 空间先验。
- **Sparse Spatial Tokens**：只在少量关键空间位置维护特征。
- **Instance Slots**：每个 slot 表示一个潜在实例，并在多帧中持续更新。

这类方法通常不追求“先把整个空间画清楚”，而是追求：

- 让每个候选实例能高效地从多相机图像中聚合证据
- 让模型直接在目标层面迭代更新位置、尺寸和朝向

#### 5.2.3 DETR3D：直接让 Query 去 3D 空间取证

`DETR3D` 是环视图 Sparse 3D 检测的重要起点之一。

它的关键思想是：

- 用一组 3D object queries 表示候选目标
- 每个 query 绑定一个 3D reference point
- 把 reference point 投影到各个相机图像中
- 在投影位置附近采样图像特征
- 再用这些特征更新 query，最终输出 3D 框

这意味着：

- 模型不需要先显式生成完整 BEV 图
- query 可以直接围绕候选目标位置向图像“取证”

它的重要意义在于：**把“3D 空间中的候选目标”作为第一类公民，而不是把它看成稠密 BEV 上的后处理产物。**

> [!TIP]
> **论文链接**：[DETR3D: 3D Object Detection from Multi-view Images via 3D-to-2D Queries](https://arxiv.org/abs/2110.06922)

#### 5.2.4 PETR / PETRv2：把 3D 位置信息编码进图像特征

`PETR` 系列的重要思路是：

- 不只是给 query 一个 reference point
- 还把 3D 位置信息以 position embedding 的形式更充分地注入图像特征

这有助于缓解一个关键问题：

> 多相机图像特征本身是 2D 的，如何让模型理解它们背后对应的 3D 位置语义？

`PETR / PETRv2` 说明了 Sparse 路线中的一个重要方向：

- 稀疏建模并不意味着完全不要空间信息
- 相反，它往往更依赖**高质量的位置编码和空间先验**

> [!TIP]
> **论文链接**：
> - [PETR: Position Embedding Transformation for Multi-View 3D Object Detection](https://arxiv.org/abs/2203.05625)
> - [PETRv2: A Unified Framework for 3D Perception from Multi-Camera Images](https://arxiv.org/abs/2206.01256)

#### 5.2.5 SparseBEV：用更稀疏的 BEV 实例表示做检测

`SparseBEV` 代表了另一种思路：

- 它并不完全放弃 BEV
- 但不再对整个 BEV 平面做大规模稠密建模
- 而是只在少量稀疏位置或实例候选周围维护 BEV 级信息

这类方法的意义是：

- 承认 BEV 空间对自动驾驶很重要
- 但希望避免 Dense BEV 的全量计算代价

因此它在思想上处于：

- “纯 query-based Sparse”
- 和 “完整 Dense BEV”

之间的过渡位置。

> [!TIP]
> **论文链接**：[SparseBEV: High-Performance Sparse 3D Object Detection from Multi-Camera Videos](https://arxiv.org/abs/2401.00944)

#### 5.2.6 Sparse4D：从单帧稀疏 Query 走向时空稀疏实例建模

`Sparse4D` 系列是这条路线里必须重点理解的一支，因为它不只是又增加了一个模型名，而是把 Sparse 路线向前推进了一大步：

> 从“单帧稀疏 query 检测”走向“时空稀疏实例建模”。

它的重要特征可以概括为：

1. **用稀疏实例表示目标，而不是只维护抽象 query。**
2. **让每个实例直接到多相机图像中做 instance-level 特征采样。**
3. **在时间维度上维护实例状态，而不是每一帧都从零开始检测。**
4. **把检测、运动信息和记忆更新耦合得更紧。**

换句话说，`Sparse4D` 的视角更像：

- 系统中始终维护着一组“潜在存在的目标实例”
- 每个实例都有自己的空间状态、特征状态与历史记忆
- 当前帧来临时，系统不是重新从整幅图里盲找目标，而是让这些实例去图像中更新证据

这比早期 query-based 检测更接近真实自动驾驶持续感知需求，因为真实系统从来不是“每一帧都完全失忆地重新看世界”。

相对 `DETR3D / PETR / SparseBEV`，`Sparse4D` 的推进点主要在于：

- **更强的时序建模**：把历史信息直接纳入实例状态更新。
- **更显式的实例级状态传播**：目标在跨帧中不是匿名 query，而是带状态的 instance slots。
- **更贴近持续感知**：有助于处理遮挡、短时缺失和运动连续性。

这也让 `Sparse4D` 成为 Sparse 路线和 Temporal 路线的重要交汇点。

> [!TIP]
> **论文链接**：
> - [Sparse4D: Multi-view 3D Object Detection with Sparse Spatial-Temporal Fusion](https://arxiv.org/abs/2211.10581)
> - [Sparse4Dv2: Recurrent Temporal Fusion for Multi-View 3D Object Detection](https://arxiv.org/abs/2306.14093)

#### 5.2.7 Sparse 路线的优势与局限

Sparse 方法的优点：

- 计算更集中在目标级候选上
- 不必先构造完整 Dense BEV
- 更容易和 query、实例记忆、时序更新结合
- 在高效检测上很有吸引力

Sparse 方法的风险：

- 对 query 初始化、reference point 和位置编码更敏感
- 全局上下文表示往往不如 Dense 完整
- 如果候选假设错了，后续可能很难补救
- 复杂遮挡和远距离弱纹理场景下，取证难度更高

可以把 Sparse 路线理解为：

> 不先“画完整地图”，而是先提出少量目标假设，再围绕这些假设向图像请求证据。

| Sparse 的优点 | Sparse 的缺点 |
| :--- | :--- |
| 计算更高效 | 更依赖先验与初始化 |
| 目标级推理直接 | 全局空间上下文较弱 |
| 适合与时序记忆结合 | 收敛和训练稳定性更敏感 |
| 有助于降低稠密建模冗余 | 候选漏掉后补救空间更小 |

### 5.3 Hybrid：稠密底座 + 稀疏精修

Hybrid 路线的核心思想是：

> 先用 Dense 空间表示提供全局几何底座，再用 Sparse 实例或 Query 完成更高效的目标级推理。

这条路线越来越重要，因为它承认了两件事：

- 纯 Dense 虽然完整，但太重。
- 纯 Sparse 虽然高效，但不一定能稳定覆盖复杂场景。

因此，更现实的方案往往是让两者配合。

#### 5.3.1 三种常见的 Hybrid 结合方式

常见的 Hybrid 结合方式包括：

1. **Dense backbone + sparse detection head**  
   先生成 BEV feature，再只用少量 query 在其中做检测。

2. **Dense BEV initialization + sparse query refinement**  
   稠密 BEV 负责提供初始空间上下文，Sparse query 负责逐步精修实例状态。

3. **Sparse candidate generation + dense local feature aggregation**  
   先生成少量候选实例，再在其局部区域聚合更丰富的稠密特征。

这类方法的共同目标是：

- 保留 Dense 的空间完备性
- 避免 Dense 的全量推理冗余
- 保留 Sparse 的目标级高效迭代

#### 5.3.2 为什么 Hybrid 很适合自动驾驶

自动驾驶的环视图 3D 检测有一个很现实的特点：

- 它不仅要检测目标
- 还要和 map、lane、occupancy、tracking 等任务共享表示

因此完全放弃 Dense 空间往往不现实；但如果所有任务都依赖全量 Dense 建模，成本又太高。

Hybrid 之所以有吸引力，就在于它可以：

- 用 Dense BEV 作为统一感知底座
- 再让 Sparse 实例机制负责更精细、更高效的目标推理

这使它在多任务和量产系统中都很有现实意义。

#### 5.3.3 Hybrid 相比纯 Dense / 纯 Sparse 的取舍

Hybrid 通常能改善：

- 目标级推理效率
- 遮挡和复杂场景下的实例精修能力
- 与时序记忆的结合灵活性

但它也会引入：

- 系统复杂度更高
- 模块耦合更强
- 调参与训练更麻烦

换句话说，Hybrid 并不是“简单叠加两者优点”，而是工程折中后的更复杂系统。

| 路线 | 空间建模完整性 | 推理效率 | 时序扩展性 | 多任务兼容性 | 实现复杂度 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **Dense** | 强 | 中到低 | 强 | 很强 | 中 |
| **Sparse** | 中到弱 | 强 | 很强 | 中 | 中到高 |
| **Hybrid** | 强 | 中到强 | 很强 | 强 | 高 |

> [!TIP]
> 理解 Hybrid 最重要的是抓住一句话：
> **它不是“Dense 不够好”的替代品，也不是“Sparse 不够强”的补丁，而是自动驾驶系统在完整空间表示和目标级高效推理之间做出的现实折中。**

## 6. 时序方案：多帧纯视觉 3D 检测是如何做的

时序方案（Temporal Modeling）不是纯视觉 3D 检测里的边角增强，而是很多高性能系统的关键组成部分。

### 6.1 为什么单帧不够

单帧纯视觉 3D 检测的问题主要包括：

- 单帧深度估计容易抖动。
- 遮挡目标在这一帧可能根本看不清。
- 远距离目标像素极少，单帧证据不足。
- 速度和运动趋势本来就天然跨帧。
- 图像噪声、曝光变化和运动模糊会让单帧结果不稳定。

因此，多帧信息能带来几类关键补偿：

- **几何补偿**：多帧提供更多观测角度和更稳定的深度线索。
- **可见性补偿**：被遮挡目标可能在历史帧中看得更清楚。
- **运动补偿**：目标速度和方向变化需要跨帧估计。
- **稳定性补偿**：历史信息可以抑制单帧抖动。

### 6.2 时序融合可以发生在哪一层

时序并不只有一种接法。按融合层级看，常见方法包括：

#### 1）图像级时序融合

- 直接对相邻帧图像或图像特征做早期融合。
- 好处是保留了更多原始信息。
- 缺点是跨视角、跨帧对齐更难，计算也较重。

#### 2）视角变换前的 feature-level 融合

- 先在图像特征层聚合历史信息，再做空间变换。
- 可以在进入 BEV 前提升图像证据稳定性。
- 但历史特征仍然处在透视视角中，对齐复杂。

#### 3）BEV-level 融合

- 先把当前帧和历史帧都变到统一 BEV 空间。
- 再对齐、融合历史 BEV feature。

这是 Dense 路线中最常见、也最直观的一种时序设计，因为：

- BEV 本身就在车辆坐标系或地平面坐标系中
- 做完 ego-motion 对齐后，跨帧融合更自然

#### 4）Query / Memory-level 融合

- 不直接累积整张 BEV 图
- 而是维护历史 query、instance slots 或 memory bank
- 当前帧到来时，只更新这些稀疏实例状态

这是 Sparse 路线尤其常见的一种做法。

#### 5）检测结果级的时序 refinement

- 先得到每帧检测结果
- 再利用时序后处理进行稳定化或 refinement

这种方法实现简单，但不如前面几类“深融合”那么彻底。

### 6.3 时序方案的几种典型技术机制

#### 6.3.1 Ego-motion 对齐与历史特征 Warping

车辆自己在运动，因此历史帧特征不能直接拿来和当前帧拼接，必须先对齐到同一坐标系。

这通常需要利用：

- 自车位姿变化
- 车辆里程计或 IMU 信息
- 历史帧到当前帧的坐标变换

可以把它理解为：

> 历史 BEV 特征只有先“转到现在的车身坐标系”，才能和当前帧可靠叠加。

#### 6.3.2 历史 BEV 特征累积

这是 Dense + Temporal 最典型的机制之一：

- 当前帧生成 BEV feature
- 历史帧 BEV feature 经对齐后一起融合
- 形成更稳定、更完整的空间表示

它的优点是：

- 空间一致性强
- 对静态背景和缓慢运动目标很有效

缺点是：

- 显存开销进一步增加
- 如果对齐不准，会出现拖影和重影

#### 6.3.3 Temporal Attention 与 Memory Bank

随着 Transformer 的使用增多，很多方法会把历史信息存入 memory：

- 历史 BEV tokens
- 历史 queries
- 历史实例特征

当前帧再通过 temporal self-attention 或 cross-attention 来读取这些历史记忆。

这种机制的优势在于：

- 历史信息读写更灵活
- 不一定要把所有历史特征都密集存下来
- 更容易和 Sparse 路线结合

#### 6.3.4 历史 Query / Instance 的继承与更新

这是 `Sparse4D` 这类方法尤其有代表性的机制。

它的基本直觉是：

- 目标实例是可以跨帧延续的
- 因此不必每一帧都重新初始化全部 query
- 可以把历史 instance 状态当作当前帧的先验

这会带来两个直接好处：

- 对遮挡目标更稳
- 对运动目标更容易保持身份连续和位置连续

但这也提出了新的难题：

- 历史实例如果漂了，错误可能被持续传播
- 当前帧与历史帧之间必须处理好运动和遮挡变化

#### 6.3.5 显式速度建模与运动补偿

很多自动驾驶数据集，如 `nuScenes`，不仅评估检测框，也关心速度误差。

因此，时序方法常常会进一步建模：

- 目标速度 $\left(v_x, v_y\right)$
- 历史位置到当前位置的更新关系
- 运动目标和静态背景的差异融合策略

这一点非常关键，因为：

- 静态道路结构适合直接累积
- 动态目标如果不做补偿就直接叠加，会产生拖影

### 6.4 时序不是第四种孤立范式，而是叠加能力层

这一点非常重要：

> Temporal 更像一种“能力层”，可以附着在 Dense、Sparse 和 Hybrid 任一路线之上。

#### Dense + Temporal

典型做法：

- 历史 BEV 特征对齐
- 多帧 BEV feature 融合

优点：

- 全局空间一致性强
- 易于支持 map、occupancy 等多任务

代价：

- 显存和时延继续上升

#### Sparse + Temporal

典型做法：

- 维护历史 queries / reference points / instance slots
- 利用历史 memory 更新当前稀疏实例状态

这类路线的关键代表之一就是 `Sparse4D`。

它说明了：

- Sparse 不一定只能做单帧
- 稀疏实例状态本身就非常适合时序继承

优点：

- 高效
- 对目标级连续感知很自然

难点：

- 跨帧身份稳定性
- 历史状态漂移
- query / instance 的长期更新可靠性

#### Hybrid + Temporal

典型做法：

- Dense memory 提供稳定场景底座
- Sparse refinement 负责当前帧实例级更新

这是非常有吸引力的一种路线，因为它兼顾：

- 场景完整性
- 目标级高效推理
- 时序连续性

### 6.5 代表方法与工程意义

从工程上看，时序方案并不是“越长越好”，而是要回答三个问题：

1. 历史信息接在哪一层最合适？
2. 如何对齐历史信息？
3. 如何避免历史错误被不断累积？

几类常见代表性模式如下：

| 路线 | 典型接法 | 核心优势 | 主要代价 |
| :--- | :--- | :--- | :--- |
| **Dense + Temporal** | 历史 BEV feature 融合 | 空间稳定、上下文完整 | 显存与延迟较高 |
| **Sparse + Temporal** | 历史 query / instance 更新 | 高效、实例连续性强 | 状态漂移风险更大 |
| **Hybrid + Temporal** | Dense memory + sparse refinement | 折中效果好 | 系统复杂度高 |

> [!TIP]
> 学习时序方案时，最重要的不是背模型名，而是先问：
> **历史信息是存成 BEV 图、存成 query，还是存成实例状态？**
> 这个选择几乎决定了方法属于 Dense、Sparse 还是 Hybrid 的哪一种时序扩展。

## 7. Transformer 与统一建模趋势

Transformer 对纯视觉 3D 检测的重要性，不只是“换了一个 backbone”，而是它提供了一种更统一的目标级和时空级建模方式。

### 7.1 Query、Reference Points 与 3D 位置先验

在纯视觉 3D 检测里，query 不再只是抽象向量，它往往会承载：

- 候选目标的空间位置先验
- 候选目标的实例特征
- 候选目标的历史记忆

这就是为什么 `DETR3D`、`PETR`、`Sparse4D` 虽然具体实现不同，但都很重视：

- query 设计
- reference points
- positional encoding

因为在没有直接点云的情况下，**位置先验本身就是模型理解 3D 空间的重要支点。**

### 7.2 多相机 Cross-Attention 如何做空间采样

Transformer 在环视图 3D 检测中的一个关键作用，是让模型学会：

- 当前实例可能在多个视角中出现
- 不同相机提供的是同一目标的不同投影视角
- 应该围绕候选 3D 位置去各个视角中取证

这正是多相机 cross-attention 最吸引人的地方：

- 它不是盲目地在全图搜索
- 而是围绕空间先验做受约束的信息聚合

### 7.3 BEV Tokens 与统一空间表征

在 Dense 或 Hybrid 路线中，Transformer 还经常被用来建模：

- BEV tokens 之间的全局关系
- BEV 与图像特征之间的交互
- 当前帧与历史帧之间的时序联系

这意味着 Transformer 既可以服务于：

- 稀疏实例更新
- 也可以服务于稠密 BEV 空间建模

从这个角度看，它反而成为了 Dense 和 Sparse 之间的桥梁。

### 7.4 为什么不同方法在底层越来越像

虽然表面上方法名字很多，但底层组件其实越来越趋同：

- query
- reference points
- positional encoding
- deformable attention
- BEV tokens
- temporal memory

这种趋同说明了一件事：

> 纯视觉 3D 检测正在从“彼此割裂的方法堆栈”，逐渐走向“统一的时空感知框架”。

在这个统一趋势里：

- Dense 路线强调统一空间底座
- Sparse 路线强调统一实例表示
- Hybrid 路线强调两者协同
- Temporal 路线强调历史记忆和状态传播

而 `Sparse4D` 很值得特别关注，因为它很好地体现了这一趋势：

- 它保留了 Sparse 路线的实例级表达
- 又把时序记忆和状态更新纳入核心框架

这让它不再只是“单帧 query 检测器”，而更像一个持续运行的时空实例感知系统。

## 8. 训练评估、模型选型与工程判断

### 8.1 3D 检测常用评估指标

纯视觉 3D 检测的评估比 2D 检测更复杂，因为它不仅关心“有没有检出来”，还关心：

- 三维位置准不准
- 朝向准不准
- 尺寸准不准
- 速度准不准

常见基础指标包括：

- **BEV IoU**：在俯视平面上的框重叠程度。
- **3D IoU**：在三维空间中的框重叠程度。
- **AP / mAP**：检测精度的总体衡量。

对于 3D 检测，IoU 的一个核心意义在于：

- BEV IoU 更直接反映地平面定位和朝向误差。
- 3D IoU 更全面，但对高度和尺寸误差也更敏感。

### 8.2 nuScenes 常用指标与它们在说什么

在自动驾驶纯视觉 3D 检测中，`nuScenes` 是非常重要的评测基准。它除了 `mAP` 之外，还常使用：

- **NDS（nuScenes Detection Score）**
- **mATE**：平均平移误差
- **mASE**：平均尺度误差
- **mAOE**：平均朝向误差
- **mAVE**：平均速度误差
- **mAAE**：平均属性误差

这套指标的意义在于：

- 它提醒我们不能只看框有没有对上
- 还要看位置、尺寸、朝向、速度是不是可靠

对自动驾驶来说，一个模型即便 `mAP` 不低，如果：

- 平移误差很大
- 朝向很飘
- 速度估计不稳

那么它对下游规划的帮助仍然有限。

### 8.3 为什么只看 AP 不够

如果一个模型的 AP 很高，但：

- 深度波动大
- 朝向不稳定
- 相邻帧之间结果抖动明显

那它在真实自动驾驶系统里仍然可能不好用。

因此纯视觉 3D 检测选型时，往往还要特别关注：

- 距离误差
- 朝向误差
- 速度误差
- 跨帧稳定性
- 时延与显存成本

### 8.4 Dense、Sparse、Hybrid、Temporal 的选型取舍

四类路线的工程取舍大致如下：

| 路线 | 优势 | 劣势 | 典型适用场景 |
| :--- | :--- | :--- | :--- |
| **Dense** | 空间完整，多任务友好 | 成本高 | BEV 为核心的统一感知系统 |
| **Sparse** | 高效，目标级推理直接 | 先验依赖强 | 高效检测、实例级建模 |
| **Hybrid** | 完整性与效率折中 | 系统更复杂 | 量产型多任务感知系统 |
| **Temporal** | 提升稳定性和速度估计 | 对齐与记忆设计复杂 | 多帧稳定感知、持续感知 |

如果从实践角度给建议，可以简单理解为：

- 想快速理解环视纯视觉 3D 检测，先学 `LSS / BEVDet / BEVDepth` 这条 Dense BEV 主线。
- 想理解 query-based 多相机 3D 检测，重点看 `DETR3D / PETR`。
- 想理解 Sparse 如何真正进入持续感知，重点看 `Sparse4D`。
- 想理解未来量产系统为什么越来越强调折中方案，重点看 Hybrid 和 Temporal 的组合逻辑。

> [!TIP]
> 对自动驾驶来说，没有任何一条路线是“绝对正确答案”。
> 更常见的真实情况是：**系统最终会根据算力预算、延迟约束、多任务需求和团队经验，在 Dense、Sparse、Hybrid、Temporal 之间做组合。**

## 9. 学习路径、资料与实践建议

### 9.1 一个推荐的学习顺序

如果你刚开始系统学习纯视觉 3D 检测，一个比较顺的路径是：

1. 先理解 3D 框参数化、坐标系和投影关系。
2. 再理解单目 3D 检测为什么病态。
3. 学会 Dense BEV 的基本范式：图像特征如何进入 BEV。
4. 学会 Sparse 路线的基本逻辑：query / reference points / instance 是怎么工作的。
5. 重点理解 `Sparse4D` 这类方法为什么能把 Sparse 和 Temporal 结合起来。
6. 再回头看 Hybrid 为什么会成为现实折中。
7. 最后系统理解多帧时序融合的设计空间。

### 9.2 推荐数据集与开源框架

学习纯视觉 3D 检测时，经常会用到以下数据集：

- **KITTI**：单目 / 双目 3D 检测经典基准，适合入门。
- **nuScenes**：环视多相机 + 时序 + 速度评估，非常适合自动驾驶研究。
- **Waymo Open Dataset**：规模大、场景丰富，工业影响力强。

常见开源框架包括：

- `MMDetection3D`
- `OpenMMLab` 生态中的 BEV 系列实现
- 各论文官方代码仓库

### 9.3 推荐论文阅读顺序

如果只想抓主线，不妨按下面顺序读：

1. `LSS`
2. `BEVDet`
3. `BEVDepth`
4. `DETR3D`
5. `PETR`
6. `Sparse4D`
7. 再回头对比 Dense、Sparse、Hybrid、Temporal 的共同点与差异

### 9.4 小结与实践建议

可以用下面几句话概括这一章最重要的内容：

1. **纯视觉 3D 检测的本质，是在没有直接深度的前提下恢复三维空间。**
2. **单目路线暴露了问题本质，环视路线则更符合自动驾驶的真实需求。**
3. **Dense 路线强调完整空间表示，Sparse 路线强调目标级高效推理，Hybrid 路线强调两者协同。**
4. **Temporal 不是附属技巧，而是纯视觉 3D 检测稳定落地的重要能力层。**
5. **Sparse4D 代表了 Sparse 路线向时空持续感知演进的重要方向。**

如果从工程落地角度给出几个最实用的建议：

- 不要只看总分，要结合平移误差、朝向误差、速度误差和跨帧稳定性。
- 不要把 Dense 和 Sparse 理解成“谁先进谁落后”，它们本质上是在不同代价下建模空间。
- 学习时优先抓“表示方式”和“信息流”，不要先被论文名字淹没。
- 想真正理解自动驾驶纯视觉 3D 检测，必须同时看单帧与多帧、空间与实例、表示与工程成本。

> [!TIP]
> **推荐阅读主线**：
> - [Lift, Splat, Shoot](https://arxiv.org/abs/2008.05711)
> - [BEVDet](https://arxiv.org/abs/2112.11790)
> - [BEVDepth](https://arxiv.org/abs/2206.10092)
> - [DETR3D](https://arxiv.org/abs/2110.06922)
> - [PETR](https://arxiv.org/abs/2203.05625)
> - [Sparse4D](https://arxiv.org/abs/2211.10581)
