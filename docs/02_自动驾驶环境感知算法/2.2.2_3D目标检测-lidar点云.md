# 2.2.2 3D目标检测-LiDAR点云

LiDAR 点云 3D 目标检测（LiDAR-based 3D Object Detection）是自动驾驶感知系统中最经典、也最成熟的三维检测路线之一。它的任务并不复杂地说，就是直接从激光雷达采集到的三维点云中找出周围目标，并恢复它们在真实空间中的位置、尺寸、朝向，必要时还包括速度等状态。

如果说 2D 检测回答的是“图像里有什么”，纯视觉 3D 检测回答的是“能不能只靠图像猜出三维空间”，那么 LiDAR 3D 检测真正要回答的问题是：

- 哪些点属于一辆车、一个行人或一个骑行者？
- 这些点支持怎样的三维包围盒？
- 这个目标在自车坐标系里到底在哪里，朝向哪里，是否在运动？

它的标准输出通常是一组 3D 框参数：

- 目标中心位置 $\left(x, y, z\right)$
- 目标尺寸 $\left(w, l, h\right)$
- 目标朝向 `yaw`
- 目标类别 `class`
- 置信度 `score`
- 可选速度 $\left(v_x, v_y\right)$

这些结果会继续送往多目标跟踪、行为预测、风险评估、规划控制等下游模块。因此，LiDAR 3D 检测长期都是自动驾驶几何感知的主力基础模块。

但这条路线真正有意思的地方在于：**LiDAR 明明已经给了三维空间信息，为什么 3D 检测仍然会发展出这么多不同的方法？**

答案也正是这一章的主线：

- 因为点云虽然在三维空间里，却是稀疏、无序、不规则的。
- 因为目标经常只有局部表面可见，而不是完整实体。
- 因为自动驾驶既要精度，也要实时性和可部署性。
- 因为下游越来越关心 BEV、跟踪和规划，所以检测器的形态也会跟着收敛。

换句话说，LiDAR 3D 检测的发展史，并不是“模型越来越复杂”，而是研究者不断在回答三个问题：

1. 如何表示原始点云？
2. 如何在精度和效率之间平衡？
3. 如何让检测结果更适合自动驾驶系统真正使用？

> [!TIP]
> **可以把这一章当成一条技术演进线来读：**
> 早期方法在想“怎么从点里找目标”，中期方法在想“怎么把点云变成卷积更擅长处理的表示”，后期方法则越来越聚焦“怎么在 BEV 上高效、稳定地输出自动驾驶真正需要的目标状态”。

---

## 1. LiDAR 3D 检测在解决什么问题

LiDAR 3D 检测最常面对的目标包括：

- 动态交通参与者：车辆、行人、骑行者、摩托车、卡车、公交车。
- 静态或半静态障碍物：锥桶、路障、施工设备、异常停靠车辆。
- 特殊结构目标：拖挂车、工程车、散落物、异形障碍物。

相比 2D 检测，它的价值更“几何直接”：

- 为跟踪提供三维空间中的初始框和姿态。
- 为预测提供更可靠的位置、朝向和速度初始化。
- 为规划控制提供可直接用于避障与安全距离计算的几何对象。
- 为占用、场景理解和高精地图对齐提供目标级结构先验。

LiDAR 路线的核心优势在于：输入本身已经位于真实三维空间，而不是二维投影平面。因此它通常在位置、尺寸和朝向估计上更稳，也对夜间、逆光等光照变化不那么敏感。

但它并不等于“问题已经解决”，因为 LiDAR 点云天生存在几类困难：

- 点云是无序点集，不是规则网格。
- 点密度分布极不均匀，近处稠密、远处稀疏。
- 目标常常只剩局部表面点，遮挡后证据更少。
- 不同线数、安装高度和扫描模式会显著改变数据分布。

于是 LiDAR 3D 检测的难点不再是“从 2D 猜 3D”，而变成了：

> 如何在有限、稀疏、局部可见的点证据下，稳定恢复目标级三维结构。

为了理解后面的方法演进，先固定几个最常见的基础对象。

### 1.1 点云、坐标系与 3D 框

LiDAR 原始输入通常是一组离散点：

$$
p_i = \left(x_i, y_i, z_i, r_i\right)
$$

其中：

- $\left(x_i, y_i, z_i\right)$ 是点在传感器坐标系下的位置。
- $r_i$ 常表示反射强度（intensity / reflectance）。

如果使用多帧点云，还可能附加：

- 时间戳偏移
- 激光线束编号
- 运动补偿后的位姿信息

LiDAR 3D 检测中最常见的坐标系包括：

- **LiDAR 坐标系**：以雷达本体为参考。
- **车辆坐标系（Ego Coordinate）**：以自车中心或后轴中心为参考。
- **全局坐标系（Global Coordinate）**：用于多帧拼接、定位和地图对齐。
- **BEV 坐标系**：把三维空间投影到地面平面的鸟瞰表达。

目标框通常参数化为：

$$
b = \left(x, y, z, w, l, h, \theta\right)
$$

其中：

- $\left(x, y, z\right)$ 是框中心位置。
- $\left(w, l, h\right)$ 是宽、长、高。
- $\theta$ 是绕垂直轴的朝向角，通常对应 `yaw`。

在 `nuScenes` 这类数据集里，还经常附带速度：

$$
b = \left(x, y, z, w, l, h, \theta, v_x, v_y\right)
$$

这些概念本身不难，但它们会决定一个方法最终更偏向点级建模、体素建模还是 BEV 建模。

### 1.2 为什么 LiDAR 检测天然会走向 BEV

虽然 LiDAR 给的是三维点，但自动驾驶真正关心的很多关系都发生在地面平面上：

- 车辆是否会相撞，首先体现在平面位置关系上。
- 跟踪和预测更习惯围绕目标中心与速度建模。
- 规划模块消费的往往也是地平面上的几何状态。

这意味着一个检测器即使在前端处理原始点云，后端也常常会逐步走向 BEV 表达。这个趋势会在后面的方法演进里不断出现。

## 2. 第一阶段：从几何规则到深度学习

LiDAR 3D 检测最早并不是直接从神经网络开始的。早期方法更像经典几何视觉和传统机器学习的延伸，它们往往依赖：

- 地面去除
- 聚类分割
- 手工几何特征
- 模板匹配或传统分类器

这种路线的直觉很自然：既然 LiDAR 已经给了三维点，那就先把点聚成团，再判断哪一团像车、哪一团像行人。

问题在于，道路场景很快把这套思路逼到边界：

- 遮挡后，同一个目标可能只剩很小一片点。
- 相邻目标之间可能彼此粘连，很难只靠聚类分开。
- 不同车辆、行人、工程车的形状变化远大于手工规则能覆盖的范围。
- 远处小目标点数太少，传统特征非常脆弱。

于是研究重点逐渐从“人来定义什么样的几何像一辆车”，转向“让网络自己学出目标级几何模式”。

这一阶段最重要的变化不是某一个具体模型，而是观念上的转折：

> LiDAR 3D 检测不再把点云看成一堆待聚类的散点，而开始把它当成可以端到端学习的空间表示。

这一步看似朴素，却决定了后面所有路线都会围绕“表示学习”展开，而不是只围绕几何规则展开。

---

## 3. 第二阶段：从原始点到体素化表示

深度学习进入 LiDAR 检测之后，第一件真正棘手的事不是分类，而是表示。

因为图像天然是规则网格，卷积很擅长；但点云不是。原始点云有三个让神经网络头疼的特征：

- 点是无序的。
- 邻接关系不是显式给出的。
- 空间很大，但大部分区域都是空的。

所以方法演进很快来到一个核心分叉：到底是直接处理原始点，还是先把点变成网络更容易处理的结构？

### 3.1 直接处理原始点：表达最自然，但成本很高

最直接的想法当然是：既然输入就是点，那就别先改写它，直接在点级做建模。

这条路线的优点非常明显：

- 更贴近原始数据，不容易过早损失几何细节。
- 对目标表面结构、边界和局部形状表达更自然。
- 理论上更适合保留细粒度几何信息。

但困难也很直接：

- 邻域构图、采样和聚合都很重。
- 大范围道路场景点数很多，推理成本会上升很快。
- 车端部署时，实时性压力会非常明显。

因此，“直接点级建模很优雅，但不够便宜”几乎成为这一阶段的共同认识。

### 3.2 体素化为什么会成为关键转折点

为了把不规则点集变成卷积友好的表示，研究者开始把三维空间离散成小单元，也就是 **voxel**。

这个想法看起来像一种工程妥协，但它之所以重要，是因为它同时解决了几个问题：

- 把无序点集变成了可索引的离散空间。
- 可以在体素内聚合局部点特征。
- 可以引入卷积 backbone 建立层次化空间特征。
- 后续还能利用稀疏卷积，只在非空区域计算。

也就是说，体素化并不是为了“让点云看起来像图像”，而是为了给高效的三维特征提取创造条件。

### 3.3 `VoxelNet`：把体素化真正做成端到端学习

`VoxelNet` 是 LiDAR 3D 检测历史里非常关键的一步。它的重要性不只在于性能，而在于它证明了：

- 可以先把点云划分成体素。
- 可以在体素内部学习点级聚合特征。
- 然后把这些特征送入三维卷积网络完成检测。

它标志着一个重要共识的形成：

> 点云不一定要原封不动地直接处理，先把它离散成合适的空间单元，再做学习，可能是更现实的路线。

当然，`VoxelNet` 也暴露出一个新的瓶颈：如果把整个三维空间都当成稠密体来卷，算力开销仍然不低。

> [!TIP]
> **论文链接**：[VoxelNet: End-to-End Learning for Point Cloud Based 3D Object Detection](https://arxiv.org/abs/1711.06396)

### 3.4 `SECOND`：为什么稀疏卷积会迅速成为主流

`SECOND` 的核心价值在于，它抓住了 LiDAR 点云最真实的一个事实：**三维空间里大部分地方本来就是空的。**

所以它引入了大规模的 **Sparse Convolution**，只在非空体素上计算。这样做的直接结果是：

- 保留了 voxel-based 表示的结构优势。
- 避免了对整块空空间做无意义卷积。
- 让 LiDAR 检测真正开始变得更接近可实时部署。

这一点非常关键，因为自动驾驶不是单纯追论文分数，而是要把模型放到持续运行的系统里。`SECOND` 之后，体素化加稀疏卷积就逐渐成为主流高性能基线。

> [!TIP]
> **论文链接**：[SECOND: Sparsely Embedded Convolutional Detection](https://pmc.ncbi.nlm.nih.gov/articles/PMC6210968/)

## 4. 第三阶段：从两阶段精修到单阶段检测

当表示问题逐渐稳定之后，另一个问题就摆到了台前：检测结构到底应该怎么设计？

这时 LiDAR 检测和 2D 检测出现了相似的分化：

- 一条路线是先找候选框，再做精修，也就是两阶段。
- 另一条路线是直接在空间特征图上密集预测，也就是单阶段。

这不是一个单纯的架构偏好问题，而是一次很典型的工程权衡。

### 4.1 两阶段路线：为什么大家会先追求精修能力

LiDAR 点云经常只有局部可见表面，所以研究者很自然会想：

- 能不能先找到一个大致位置？
- 再围绕这个候选区域做更细的几何推理？

这正是两阶段方法的直觉。它们通常更强调：

- proposal 质量
- 候选区域内的局部点特征提取
- 框中心、尺寸和朝向的进一步精修

因为局部点结构会被看得更仔细，所以两阶段方法长期在高精度定位上很有吸引力。

### 4.2 `PointRCNN`：直接从点里提出候选框

`PointRCNN` 是两阶段 LiDAR 检测中的代表工作之一。它非常有代表性，因为它把“点级建模”和“两阶段 proposal 精修”结合了起来。

它的思路可以概括成三步：

1. 先在原始点上做前景分割。
2. 围绕前景点生成 3D proposal。
3. 再对 proposal 内的点做精修回归。

它说明了一件很重要的事：

> 即使不先把点云彻底规整成网格，也可以直接在点级结构上完成候选框生成和局部精修。

但它也把成本问题暴露得很充分：

- 点级邻域计算仍然偏重。
- proposal 管理和 RoI 精修链路更长。
- 大场景下实时性通常不如更轻量的单阶段方案。

> [!TIP]
> **论文链接**：[PointRCNN: 3D Object Proposal Generation and Detection from Point Cloud](https://arxiv.org/abs/1812.04244)

### 4.3 `Part-A2` 与 `PV-RCNN`：为什么“点和体素结合”会变得有吸引力

随着研究继续推进，大家开始发现一个现实情况：

- 纯点级精修几何能力强，但太重。
- 纯体素 backbone 很高效，但局部细节可能不够细。

于是就出现了“点和体素结合”的思路。

`Part-A2` 更强调 proposal 内的部件级结构建模，而 `PV-RCNN` 则给出了一条非常经典的折中路线：

- 用体素卷积提取高效的全局空间特征。
- 用关键点或局部点特征做细粒度精修。

`PV-RCNN` 的意义不是简单叠了两个模块，而是它非常清楚地展示了：

> 全局结构更适合交给体素 backbone，局部边界和精修更适合交给点级特征。

这类方法把“两阶段高上限”的优势留住了一部分，但也意味着结构复杂度和实现复杂度继续上升。

> [!TIP]
> **论文链接**：
> - [Part-A^2 net: 3D Part-Aware and Aggregation Neural Network for Object Detection from Point Cloud](https://arxiv.org/abs/1907.03670)
> - [PV-RCNN: Point-Voxel Feature Set Abstraction for 3D Object Detection](https://arxiv.org/abs/1912.13192)

### 4.4 为什么工业界越来越偏向单阶段

两阶段方法的上限确实很吸引人，但自动驾驶的工程现实很快给出了一条清晰约束：

- 检测不是离线任务，而是持续运行的在线任务。
- 下游往往还要并行跑跟踪、预测、占用、地图等模块。
- 系统更在意整体延迟、显存、吞吐和部署稳定性。

这时单阶段路线的吸引力就会越来越强，因为它通常意味着：

- 更短的推理链路
- 更简单的结构
- 更稳定的部署成本
- 更容易和其他 BEV 任务共享 backbone

于是研究方向逐渐从“如何把 proposal 精修做得更强”，转向“如何让单阶段也足够准，而且更适合量产系统”。

## 5. 第四阶段：从完整 3D 处理走向 BEV 主导

当研究走到这里，LiDAR 3D 检测出现了一个越来越明显的收敛趋势：**很多方法前端看起来各不相同，但后端越来越喜欢在 BEV 上做目标预测。**

这背后不是时髦，而是任务本身决定的。

### 5.1 为什么很多方法最后都会落到 BEV

对自动驾驶来说，BEV 有几个非常现实的优势：

- 目标之间最关键的相对关系发生在地平面。
- 旋转框在 BEV 中定义自然。
- 跟踪和预测更容易围绕目标中心与速度建模。
- 地图、占用和规划也大多工作在 BEV 语义下。

所以无论前端是点、体素、pillar 还是其他表示，后端都很容易逐步汇聚到同一个问题：

> 如何在一个统一的地平面空间里，稳定地输出目标中心、尺寸、朝向和速度。

### 5.2 `PointPillars`：为什么“早点变成 BEV”这么有吸引力

`PointPillars` 的做法很有代表性。它没有继续坚持完整三维体素，而是把空间沿高度方向压缩成柱状单元 `pillars`，尽快转成二维 BEV 特征图。

这个设计很讨巧，因为它直接换来了一整套工程优势：

- 预处理相对简单。
- 检测流程更接近成熟的 2D CNN 工具链。
- 推理速度快，部署友好。

它的代价当然也很明确：

- 高度维细节被更早压缩。
- 对复杂垂向结构的表达不如完整体素细。

但在自动驾驶里，这种交换常常是划算的，因为大多数目标的关键几何关系本来就主要体现在地平面上。

> [!TIP]
> **论文链接**：[PointPillars: Fast Encoders for Object Detection from Point Clouds](https://arxiv.org/abs/1812.05784)

### 5.3 `CenterPoint`：为什么 Center-based BEV 会成为重要主流

如果说 `PointPillars` 代表的是“尽早转 BEV”，那么 `CenterPoint` 代表的则是“在 BEV 上用更自然的目标状态表达做检测”。

它的基本思路是：

- 在 BEV 平面上预测目标中心热力图。
- 再回归尺寸、高度、朝向和速度等属性。

这个设计会受到欢迎，不只是因为它省掉了一部分 anchor 设计，而是因为它和自动驾驶的目标状态定义非常一致：

- 目标在 BEV 中确实有物理意义明确的中心。
- 运动目标的状态也天然围绕中心演化。
- 跟踪和预测可以更自然地接在后面。

因此，`CenterPoint` 之后，LiDAR 检测里“center-based + BEV + 多任务扩展”这条路线的吸引力就变得非常强。

> [!TIP]
> **论文链接**：[CenterPoint: 3D Object Detection and Tracking from Point Clouds](https://arxiv.org/abs/2006.11275)

### 5.4 这一步演进真正说明了什么

从完整 3D 体空间走向 BEV 主导，并不意味着三维信息不重要，而是说明研究者逐渐接受了一个现实：

- 高度信息仍然重要，但更像局部属性。
- BEV 关系则往往决定了检测、跟踪和规划的全局组织方式。

所以今天很多主流系统都可以理解为：

- 前端用不同方式从点云里提取三维几何。
- 中后端则越来越统一地在 BEV 上组织目标级推理。

## 6. 代表模型串讲：从 `VoxelNet` 到 `CenterPoint`

如果把前面的演进压缩成一条最小阅读主线，可以大致这样看：

### 6.1 `VoxelNet`

- 它解决的问题：点云太不规则，不能直接高效卷积。
- 它给出的答案：先体素化，再在体素内学习特征。
- 它的历史意义：把 voxel-based 端到端学习正式立住。

### 6.2 `SECOND`

- 它解决的问题：体素化后依然太贵。
- 它给出的答案：只在非空体素上做稀疏卷积。
- 它的历史意义：让 voxel-based 路线真正具备更强工程可行性。

### 6.3 `PointPillars`

- 它解决的问题：完整三维体素仍然不够轻。
- 它给出的答案：沿高度压缩成 pillars，尽快转成 BEV。
- 它的历史意义：把 LiDAR 检测大幅推向更高效、更部署友好的方向。

### 6.4 `PointRCNN`

- 它解决的问题：如何直接在原始点上做候选框生成与精修。
- 它给出的答案：前景点分割 + proposal + 点级 refinement。
- 它的历史意义：证明点级两阶段路线可以做到很强的几何精修。

### 6.5 `PV-RCNN`

- 它解决的问题：纯点级太重，纯体素细节不够。
- 它给出的答案：体素 backbone 提供全局效率，点级关键点提供局部精修。
- 它的历史意义：展示了 point-voxel 混合路线的强大折中价值。

### 6.6 `CenterPoint`

- 它解决的问题：如何在高效 BEV 表示上得到更自然的目标状态建模。
- 它给出的答案：center-based BEV 检测与状态回归。
- 它的历史意义：把 LiDAR 检测和后续跟踪、多任务 BEV 系统连接得更顺。

如果只看这些模型名，容易觉得它们彼此只是“换了个网络”。但放在演进链里看，会发现每一步都在回答一个很具体的问题：

- 点云怎么表示？
- 计算怎么降下来？
- 局部精修要不要保留？
- 最终状态如何更自然地对接自动驾驶系统？

## 7. 今天的主流格局：Point / Voxel / Pillar 各自站在哪里

演进走到今天，LiDAR 检测并没有收敛成唯一答案，而是形成了几种比较稳定的主流范式。它们不是平行罗列的知识点，而是不同历史取舍的沉淀结果。

### 7.1 Point-based：保留原始几何细节

Point-based 方法尽量少做离散化，直接对原始点或局部邻域建模。

它们通常更擅长：

- 捕捉目标表面几何细节
- 保留局部边界信息
- 做 proposal 内的精细结构推理

但问题也很持续：

- 邻域建模成本高
- 大场景推理压力大
- 纯点级方案不太容易成为量产系统的最朴素起点

所以这条路线今天更常见的角色是：

- 高精度研究方案
- 局部精修模块
- 与体素 backbone 结合的混合方案组成部分

### 7.2 Voxel-based：精度与效率之间最稳健的折中

Voxel-based 方法把空间切成小体素，再利用稀疏卷积建模。

它们之所以长期主流，是因为几乎正好踩在自动驾驶最需要的平衡点上：

- 比原始点级更规整，适合 backbone 建模。
- 比 pillar 更保留三维几何结构。
- 稀疏卷积又让效率不至于完全失控。

这条路线常常被视为：

- 主流高性能基线
- 研究和工程都能接受的稳健折中

### 7.3 Pillar-based：尽快进入 BEV 的工程方案

Pillar-based 方法本质上是在说：

> 既然最终很多任务都围绕地平面展开，那就不如更早把高度维压缩掉，尽快转成 BEV 问题。

它们的优点非常明确：

- 速度快
- 实现成熟
- 部署友好

代价也很明确：

- 早期就牺牲了一部分垂向细节
- 对复杂高度结构的表达天然更弱

因此它通常很适合：

- 实时部署
- 快速建立可靠基线
- 对系统总时延要求更严格的场景

### 7.4 Range View：始终存在，但不是主流收敛方向

Range View 会把点云投影成“雷达图像”，然后复用 2D CNN。

它的优点是：

- 速度快
- 2D CNN 工具链成熟

它的局限也长期存在：

- 投影后邻接关系会扭曲
- 遮挡混叠更难处理
- 物理空间中的近邻不一定仍然在图上接近

因此，Range View 一直是一个很有价值的分支，但并没有像 voxel-BEV 路线那样成为更广泛的主流收敛点。

### 7.5 一张表看清这些路线的区别

| 路线 | 核心思路 | 优势 | 劣势 | 更常见的角色 |
| :--- | :--- | :--- | :--- | :--- |
| **Point-based** | 直接建模点集 | 几何细节保真度高 | 计算重，部署复杂 | 精修、高精度研究 |
| **Voxel-based** | 体素化 + 稀疏卷积 | 表达和效率平衡好 | 体素大小等超参敏感 | 主流高性能基线 |
| **Pillar-based** | 高度压缩 + BEV | 快、成熟、部署友好 | 垂向细节损失更早 | 实时部署、量产起点 |
| **Range View** | 投影为雷达图像 | 速度快、2D CNN 友好 | 投影扭曲、遮挡混叠 | 特定传感器或速度优先方案 |

## 8. 工程视角：自动驾驶里真正关心什么

如果只看论文脉络，很容易把 LiDAR 3D 检测理解成“谁的 AP 更高”。但系统真正上线时，更关键的问题通常不是榜单，而是下面这些东西。

### 8.1 只看 AP 远远不够

常见评估指标包括：

- **3D AP**：基于 3D 框 IoU 的平均精度。
- **BEV AP**：基于 BEV 投影框 IoU 的平均精度。
- **Recall**：真实目标中被召回的比例。
- **mAP**：多类别 AP 的平均值。

在 `KITTI` 里，人们常看 Easy / Moderate / Hard 以及 3D AP、BEV AP；在 `nuScenes` 里，则还会关心：

- `NDS`
- 平移误差
- 尺度误差
- 朝向误差
- 速度误差

这背后的原因很简单：一个模型即使总分不低，也可能在实际系统里很难用，比如：

- 远距离行人召回差
- 小障碍物漏检明显
- 框中心还行，但朝向老是抖
- 速度估计不稳定，影响跟踪和预测

### 8.2 自动驾驶更在意哪些失败模式

在车端，真正让人头疼的通常不是“平均分低一点”，而是这些具体失败模式：

- 远距离目标点太少，直接漏检。
- 遮挡后只剩局部表面，框体恢复漂移。
- 小目标尺寸估计抖动，导致跟踪状态不稳。
- 稀疏噪点被误报成障碍物，引发规划保守。
- 车辆框朝向错得很离谱，导致下游运动推理异常。

也正因为如此，很多工程判断和论文排序并不完全一致。一个系统可能不会选理论上最强的模型，而会选：

- 延迟更低的
- 速度估计更稳的
- 多任务更容易整合的
- 在长尾场景下更少出怪问题的

### 8.3 为什么 BEV 与单阶段在工程上越来越吃香

从工程角度看，BEV 和单阶段越来越主流，不是因为它们总是绝对最优，而是因为它们同时满足了几个现实条件：

- 表示统一，便于和跟踪、预测、占用、地图共享。
- 推理链路短，更容易控制时延。
- 部署成本更清晰，更容易稳定复现。
- 目标状态表达和下游接口天然一致。

这也是为什么很多团队的实践起点往往不是最复杂的两阶段模型，而是：

- `PointPillars`
- `SECOND`
- `CenterPoint`

这些方法未必在每个维度都极致，但它们很适合拿来搭建“可理解、可调、可部署”的第一条系统主线。

## 9. 学习路径与实践建议


1. 在 `OpenPCDet` 里跑通一个 `PointPillars` 或 `CenterPoint` 基线。
2. 可视化近距、远距、遮挡、行人、小障碍物几类典型样例。
3. 对照本文主线去判断问题究竟出在：
   点云太稀、表示选择不合适、局部精修不够、BEV 压缩损失过多，还是数据与标签本身限制了上限。

**思考下面问题：**

- 为什么这个方法会这样设计？
- 它到底是在解决表示问题、效率问题，还是工程对接问题？
- 它适不适合自动驾驶系统真正关心的场景和约束？
